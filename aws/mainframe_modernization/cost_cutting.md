## Executive Summary
AWS Mainframe Modernization provides pathways to migrate legacy mainframe applications (COBOL, PL/I, JCL) to the cloud through automated refactoring (AWS Blu Age) or replatforming (Micro Focus). With the managed runtime sunset for new customers, the focus has shifted toward AWS Transform and self-managed EC2 runtimes. This playbook outlines 18 targeted strategies to optimize costs associated with mainframe transformation, ongoing runtime operations, underlying compute, and legacy data storage. Key savings opportunities exist in accelerating the transition to cloud-native architectures (Java/Containers), rightsizing self-managed EC2, and implementing aggressive scheduling for non-production environments.

## Strategy Categories
### 1. Waste Elimination

#### M2-01. Remove Abandoned Mainframe Transform Environments
- **What:** Identify and delete orphaned or inactive AWS Transform for mainframe runtime environments.
- **Why It Saves Money:** AWS Transform Core is billed continuously at $0.31 per CPU core-hour ($226.30/core/month), even if the environment is idle.
- **Implementation Steps:**
  1. Audit active Transform environments via the AWS Console or CLI.
  2. Map environments to active development or testing projects.
  3. Terminate environments that have no associated project or recent activity.
- **Estimated Savings:** 100% of the cost for the abandoned environment.
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Environment tagging strategy.

#### M2-02. Delete Unattached Storage Volumes from Self-Managed Runtimes
- **What:** Clean up unattached EBS volumes that were left behind when terminating EC2 instances used for self-managed mainframe runtimes.
- **Why It Saves Money:** EBS volumes are billed provisioned per GB-month regardless of attachment status (e.g., $0.08/GB-month for gp3).
- **Implementation Steps:**
  1. Use AWS Config or custom scripts to find EBS volumes in `available` state.
  2. Snapshot the volumes if retention is needed.
  3. Delete the unattached volumes.
- **Estimated Savings:** 100% of the unattached storage cost.
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Backup policy for orphaned data.

#### M2-03. Clean Up Stale Mainframe Transformation Logs
- **What:** Implement lifecycle policies on CloudWatch Log Groups generated by mainframe transformation tools and runtimes.
- **Why It Saves Money:** CloudWatch Logs storage costs $0.03 per GB per month. Mainframe traces can generate massive volumes of log data.
- **Implementation Steps:**
  1. Identify CloudWatch Log Groups associated with Mainframe Modernization.
  2. Set retention periods (e.g., 14 days for dev, 90 days for prod) instead of the default 'Never Expire'.
- **Estimated Savings:** 50-80% of log storage costs over time.
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Compliance guidelines for log retention.

### 2. Rightsizing

#### M2-04. Rightsize Self-Managed EC2 Mainframe Runtimes
- **What:** Downsize EC2 instances running Micro Focus or custom emulators based on actual CPU and memory utilization.
- **Why It Saves Money:** Halving the instance size (e.g., from `m5.2xlarge` to `m5.xlarge`) reduces compute costs by exactly 50%.
- **Implementation Steps:**
  1. Review AWS Compute Optimizer recommendations for the specific EC2 instances.
  2. Analyze memory and CPU peaks during month-end batch processing.
  3. Change the instance type during a scheduled maintenance window.
- **Estimated Savings:** 30-50% on self-managed compute.
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** 14+ days of CloudWatch metrics.

#### M2-05. Optimize Core Count for AWS Transform Runtimes
- **What:** Deploy AWS Transform runtimes with the minimum required vCPU cores for the workload phase (e.g., 2 cores for dev vs. 16 for prod).
- **Why It Saves Money:** Billed linearly at $0.31 per core-hour. Dropping from 4 cores to 2 cores saves $452.60 per month.
- **Implementation Steps:**
  1. Monitor performance of the transformation runtime.
  2. Redeploy or adjust the runtime environment to a lower core count if utilization is consistently low.
- **Estimated Savings:** 25-50% per optimized environment.
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Performance testing baseline.

#### M2-06. Rightsize Underlying Mainframe Databases
- **What:** Downsize Amazon RDS or Aurora instances used as the datastore for the replatformed mainframe applications.
- **Why It Saves Money:** Database compute often makes up the largest portion of replatforming costs. Smaller instances linearly reduce costs.
- **Implementation Steps:**
  1. Check database CPU, memory, and IOPS utilization.
  2. Downsize RDS/Aurora instances or adjust Aurora Serverless v2 capacity ranges.
- **Estimated Savings:** 20-40% of database costs.
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Database load testing.

### 3. Commitment Discounts

#### M2-07. Purchase Compute Savings Plans for Self-Managed Runtimes
- **What:** Apply a 1-year or 3-year Compute Savings Plan to the EC2 instances powering self-managed mainframe runtimes.
- **Why It Saves Money:** Up to 72% discount off On-Demand EC2 rates in exchange for a committed hourly spend.
- **Implementation Steps:**
  1. Ensure instance rightsizing (M2-04) is complete.
  2. Analyze baseline compute usage via AWS Cost Explorer SP recommendations.
  3. Purchase a Compute Savings Plan.
- **Estimated Savings:** 20-72% on compute costs.
- **Risk Level:** Low
- **Implementation Scope:** FinOps Team
- **Prerequisites:** Stable production workload, rightsizing complete.

#### M2-08. Purchase RDS/Aurora Reserved Instances
- **What:** Commit to 1- or 3-year Reserved Instances for the databases backing the modernized mainframe application.
- **Why It Saves Money:** Up to 60% savings compared to On-Demand database rates.
- **Implementation Steps:**
  1. Finalize the database engine (e.g., PostgreSQL) and instance families.
  2. Review AWS Cost Explorer RI recommendations.
  3. Purchase standard RIs for steady-state databases.
- **Estimated Savings:** 30-60% on database compute.
- **Risk Level:** Low
- **Implementation Scope:** FinOps Team
- **Prerequisites:** Stable database architecture.

### 4. Architecture Changes

#### M2-09. Refactor COBOL to Cloud-Native Java (AWS Blu Age Pattern)
- **What:** Use automated refactoring to convert legacy COBOL into standard Java/Spring Boot code instead of maintaining emulation runtimes.
- **Why It Saves Money:** Eliminates specialized mainframe runtime or legacy IBM licensing fees. Allows execution on cheaper, standard compute (EKS/Fargate/Graviton).
- **Implementation Steps:**
  1. Assess the legacy codebase using AWS Mainframe Modernization tools.
  2. Execute the AWS Blu Age automated refactoring pipeline.
  3. Deploy the resulting Java artifacts to standard AWS compute.
- **Estimated Savings:** 70–90% infrastructure reduction.
- **Risk Level:** High
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Deep understanding of legacy business logic.

#### M2-10. Migrate Refactored Java Workloads to AWS Graviton
- **What:** Move the newly refactored Java/Spring Boot applications to run on ARM-based AWS Graviton processors (e.g., `m7g`, `t4g`).
- **Why It Saves Money:** Graviton instances offer up to 20% lower cost and up to 40% better price-performance than comparable x86 instances.
- **Implementation Steps:**
  1. Compile and test the Java application on an ARM architecture.
  2. Update EKS node groups, ECS task definitions, or EC2 Auto Scaling groups to use Graviton instance types.
- **Estimated Savings:** 20% on raw compute costs.
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Refactored Java architecture (M2-09).

#### M2-11. Modernize Batch Jobs to AWS Batch or Step Functions
- **What:** Instead of running monolithic batch jobs on 24/7 EC2 instances, refactor them into event-driven workflows using AWS Batch or Step Functions.
- **Why It Saves Money:** You only pay for the compute consumed during the batch job execution (via Fargate or spot instances), rather than paying for idle time.
- **Implementation Steps:**
  1. Analyze mainframe JCL (Job Control Language) schedules.
  2. Rewrite batch orchestration using AWS Step Functions.
  3. Execute batch processes on AWS Batch.
- **Estimated Savings:** 40-70% on batch compute costs.
- **Risk Level:** High
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Automated refactoring complete.

#### M2-12. Offload Mainframe Reporting to Amazon Redshift
- **What:** Replicate mainframe data to S3 or Redshift for complex reporting, rather than querying the transactional mainframe database.
- **Why It Saves Money:** Reduces the compute load on the expensive, heavily-licensed mainframe runtime or its primary relational database.
- **Implementation Steps:**
  1. Set up AWS DMS or a CDC tool to stream data to Amazon S3.
  2. Query data using Amazon Athena or load it into Amazon Redshift.
- **Estimated Savings:** 20-30% reduction in primary database costs.
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Data replication pipeline.

### 5. Scheduling & Auto-Scaling

#### M2-13. Schedule Non-Production Mainframe Runtime Teardowns
- **What:** Stop or delete non-production dev/staging mainframe runtime environments during nights and weekends.
- **Why It Saves Money:** Environments running 24/7 cost ~3x more than environments running only 40 hours a week (business hours).
- **Implementation Steps:**
  1. Tag dev/test environments appropriately.
  2. Use AWS Instance Scheduler (for EC2) or Lambda scripts (for Transform environments) to stop/delete resources at 6 PM and restart at 8 AM.
- **Estimated Savings:** 65-70% on non-production compute.
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Automated IaC deployments for easy recreation.

#### M2-14. Implement Auto-Scaling for Refactored Microservices
- **What:** Configure Horizontal Pod Autoscaler (EKS) or Target Tracking Scaling (ECS/EC2) for refactored Java applications.
- **Why It Saves Money:** Matches compute capacity to real-time transaction volume instead of over-provisioning for peak mainframe MIPS usage.
- **Implementation Steps:**
  1. Define CPU/Memory utilization targets (e.g., 70%).
  2. Configure auto-scaling rules.
  3. Test scale-up and scale-down behaviors during simulated load.
- **Estimated Savings:** 30-50% on production compute.
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Refactored stateless architecture (M2-09).

#### M2-15. Scale Down Database Capacity During Off-Peak
- **What:** Use Amazon Aurora Serverless v2 for replatformed databases to automatically scale compute (ACUs) down when the mainframe application is idle.
- **Why It Saves Money:** Scales down to minimal ACUs during nights/weekends instead of paying for peak provisioned capacity 24/7.
- **Implementation Steps:**
  1. Migrate the underlying datastore to Aurora Serverless v2.
  2. Define minimum and maximum ACU limits.
- **Estimated Savings:** 40-60% on database costs for variable workloads.
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Compatible database engine.

### 6. Pricing Model Optimization

#### M2-16. Leverage Spot Instances for Refactored Batch Processing
- **What:** Run fault-tolerant, refactored batch jobs on EC2 Spot Instances via AWS Batch.
- **Why It Saves Money:** Spot instances offer up to 90% discounts over On-Demand pricing.
- **Implementation Steps:**
  1. Identify idempotent, interruptible batch jobs (formerly JCL).
  2. Configure AWS Batch compute environments to use Spot.
  3. Implement retry logic for Spot interruptions.
- **Estimated Savings:** 70-90% on batch compute costs.
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Jobs must be stateless and interruptible.

#### M2-17. Optimize Mainframe Backup Storage Tiers
- **What:** Move long-term mainframe data backups and historical transaction logs to Amazon S3 Glacier Deep Archive.
- **Why It Saves Money:** S3 Standard costs $0.023/GB, whereas Glacier Deep Archive costs $0.00099/GB—a 95% reduction.
- **Implementation Steps:**
  1. Define backup retention requirements.
  2. Create an S3 Lifecycle Rule to transition older backups to Glacier Deep Archive.
- **Estimated Savings:** 95% on long-term backup storage.
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Understanding of data retrieval SLAs.

### 7. Network & Data Transfer Optimization

#### M2-18. Use VPC Endpoints for S3 Data Transfers
- **What:** Route traffic from mainframe runtimes to AWS services (like S3 for backups) through VPC Gateway Endpoints instead of NAT Gateways.
- **Why It Saves Money:** NAT Gateways charge $0.045 per GB processed. VPC Gateway Endpoints for S3 are free.
- **Implementation Steps:**
  1. Create a VPC Gateway Endpoint for S3 in the VPC.
  2. Update route tables to route S3 traffic to the endpoint.
- **Estimated Savings:** 100% of NAT Gateway data processing fees for S3 traffic.
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** VPC routing access.

## Cross-Service Synergies
- **Modernization + Analytics:** Refactoring COBOL to Java (M2-09) naturally synergizes with offloading reporting to Redshift (M2-12), as the data is already liberated from proprietary VSAM or DB2 mainframe formats, compounding compute savings.
- **Rightsizing + Commitments:** Always rightsize EC2 instances (M2-04) before purchasing Savings Plans (M2-07) to avoid over-committing to wasted capacity.
