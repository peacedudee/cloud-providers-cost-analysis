## Executive Summary
AWS Cost Explorer is a foundational FinOps tool that provides 100% free web console access for visualizing and analyzing AWS spend. While the console is free, programmatic access via the Cost Explorer API incurs charges ($0.01 per request), and enabling granular hourly data adds a fee per usage record. This playbook covers both optimizing the costs generated by Cost Explorer itself (such as heavy API polling) and leveraging Cost Explorer's reporting to uncover massive cost-savings opportunities across your broader AWS infrastructure. By moving programmatic reporting to CUR 2.0 and actively utilizing Cost Explorer's recommendations, organizations can drastically cut their AWS bill.

## Strategy Categories

### 1. Waste Elimination

#### CE-1. Replace Cost Explorer API with CUR + Athena
- **What:** Transition from querying the programmatic Cost Explorer API to analyzing AWS Cost and Usage Reports (CUR 2.0) using Amazon Athena.
- **Why It Saves Money:** The CE API costs $0.01 per request. A dashboard updating every 5 minutes across 10 custom views can cost over $4,300/month. Athena charges only ~$5/TB of data scanned, slashing reporting costs by over 95%.
- **Implementation Steps:**
  1. Enable CUR 2.0 delivery (in Parquet format) to an Amazon S3 bucket.
  2. Configure AWS Glue to catalog the CUR data.
  3. Update internal FinOps dashboards to query Athena instead of calling the `GetCostAndUsage` API.
- **Estimated Savings:** 95-99% on FinOps tooling costs
- **Risk Level:** Low
- **Implementation Scope:** FinOps Team | Engineer/DevOps
- **Prerequisites:** S3, Athena, internal dashboard tools

#### CE-2. Disable Hourly and Resource-Level Granularity for High-Scale Fleets
- **What:** Turn off hourly/resource-level granularity in Cost Explorer for accounts that generate millions of short-lived resource records (e.g., ephemeral Kubernetes pods or ECS tasks).
- **Why It Saves Money:** Hourly granularity bills at $0.00000033 per usage record per day. For thousands of ephemeral containers, these micro-charges scale up to hundreds of dollars per month unnecessarily.
- **Implementation Steps:**
  1. Open the Cost Explorer settings in the Billing console.
  2. Evaluate if resource-level or hourly data is strictly required.
  3. Uncheck "Hourly and Resource Level Data" for accounts where daily aggregates suffice.
- **Estimated Savings:** 100% on granular telemetry fees
- **Risk Level:** Low
- **Implementation Scope:** FinOps Team
- **Prerequisites:** Administrator access to Billing Console

#### CE-3. Implement Caching for Cost Explorer API Dashboards
- **What:** If you must use the CE API, implement a caching layer in your application to prevent duplicate consecutive queries for identical timeframes.
- **Why It Saves Money:** Reduces duplicate $0.01 API calls by serving historical or daily data directly from a local cache (like Redis or DynamoDB).
- **Implementation Steps:**
  1. Identify the FinOps script calling the Cost Explorer API.
  2. Implement an in-memory or database cache with a TTL of 12-24 hours for daily metrics.
  3. Route all dashboard queries through the cache before hitting the AWS API.
- **Estimated Savings:** 50-80% of CE API costs
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Caching infrastructure (Redis/Memcached/DynamoDB)

#### CE-4. Find and Terminate Unattached EBS Volumes
- **What:** Use Cost Explorer filtering to identify spend on EBS volumes (Usage Type: `EBS:VolumeUsage`) without corresponding active EC2 instance running costs in the same environment.
- **Why It Saves Money:** Unattached gp3 volumes cost $0.08/GB-month. Deleting 10 TB of orphaned volumes saves $800/month.
- **Implementation Steps:**
  1. Filter Cost Explorer by Service: `EC2-Other` and Usage Type: `Volume`.
  2. Cross-reference high volume costs with low EC2 run costs in specific regions/accounts.
  3. Snapshot and delete unattached volumes via the EC2 console.
- **Estimated Savings:** 10-15% of EC2-Other costs
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps | FinOps Team
- **Prerequisites:** EC2 Read/Write access

#### CE-5. Discover and Release Unused Elastic IPs
- **What:** Use Cost Explorer to identify charges for unattached Elastic IPs (EIPs).
- **Why It Saves Money:** AWS charges $0.005 per hour (approx. $3.60/month) for *all* Elastic IPs, even when attached, but especially punishes idle ones. Releasing 100 idle EIPs saves $360/month.
- **Implementation Steps:**
  1. Filter Cost Explorer by Service: `VPC` and Usage Type: `ElasticIP:IdleAddress`.
  2. Navigate to the VPC Console and locate unassociated EIPs.
  3. Release them back to the AWS pool.
- **Estimated Savings:** 100% of idle EIP costs
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** VPC administration access

#### CE-6. Identify and Delete Obsolete Snapshots
- **What:** Analyze EBS snapshot costs over time in Cost Explorer to detect unmanaged exponential growth of backups.
- **Why It Saves Money:** EBS snapshots cost $0.05/GB-month. Retaining years of daily snapshots without a lifecycle policy incurs massive storage fees.
- **Implementation Steps:**
  1. Filter Cost Explorer by Service: `EC2-Other` and Usage Type Group: `EBS:SnapshotUsage`.
  2. Identify accounts with rapidly scaling snapshot costs.
  3. Implement AWS Data Lifecycle Manager (DLM) to automatically delete snapshots older than 30-90 days.
- **Estimated Savings:** 40-60% of snapshot costs
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** AWS Data Lifecycle Manager

### 2. Rightsizing

#### CE-7. Identify Underutilized EC2 Instances
- **What:** Use Cost Explorer's native "Rightsizing Recommendations" tool to find EC2 instances with low CPU and memory utilization.
- **Why It Saves Money:** Downsizing an `m5.4xlarge` ($0.768/hr) to an `m5.2xlarge` ($0.384/hr) instantly cuts the instance cost by 50%.
- **Implementation Steps:**
  1. Navigate to Cost Explorer -> Rightsizing Recommendations.
  2. Review the list of instances marked as "Idle" or "Underutilized".
  3. Schedule a maintenance window to resize the instances or terminate idle ones.
- **Estimated Savings:** 20-50% on targeted EC2 instances
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps | FinOps Team
- **Prerequisites:** CloudWatch agent (for memory metrics)

#### CE-8. Rightsize Overprovisioned RDS Instances
- **What:** Analyze Amazon RDS costs in Cost Explorer to spot high baseline spend, then cross-reference with CloudWatch to rightsize databases.
- **Why It Saves Money:** Halving an RDS instance size (e.g., `db.r6g.4xlarge` to `db.r6g.2xlarge`) saves 50% on hourly database compute costs.
- **Implementation Steps:**
  1. Filter CE by Service: `RDS` and group by Instance Type.
  2. Correlate with CloudWatch RDS CPU/Memory/IOPS metrics.
  3. Modify the RDS instance type during a scheduled downtime window.
- **Estimated Savings:** 30-50% on RDS compute
- **Risk Level:** High
- **Implementation Scope:** Engineer/DevOps (DBA)
- **Prerequisites:** RDS modify permissions

### 3. Commitment Discounts

#### CE-9. Purchase Compute Savings Plans
- **What:** Use the "Savings Plans Recommendations" section in Cost Explorer to commit to a 1- or 3-year baseline compute spend.
- **Why It Saves Money:** Compute Savings Plans apply automatically across EC2, Fargate, and Lambda, providing up to 66% discounts over On-Demand pricing.
- **Implementation Steps:**
  1. Open Cost Explorer -> Savings Plans -> Recommendations.
  2. Select Compute Savings Plan, 1 or 3 years, and No/Partial Upfront.
  3. Review the recommended hourly commitment and purchase the plan.
- **Estimated Savings:** 20-66% of compute costs
- **Risk Level:** Medium (Requires financial commitment)
- **Implementation Scope:** FinOps Team | Procurement/Leadership
- **Prerequisites:** Predictable baseline compute usage

#### CE-10. Purchase EC2 Instance Savings Plans
- **What:** Use Cost Explorer to identify stable EC2 usage within specific instance families and regions to purchase EC2 Instance Savings Plans.
- **Why It Saves Money:** Provides the deepest possible discount (up to 72%), but is locked to a specific instance family and region (e.g., M5 in us-east-1).
- **Implementation Steps:**
  1. Navigate to CE Savings Plans Recommendations.
  2. Switch the recommendation parameter to "EC2 Instance Savings Plans".
  3. Validate that your architecture won't migrate families soon, then purchase.
- **Estimated Savings:** 30-72% of specific EC2 family costs
- **Risk Level:** High (Less flexible than Compute SP)
- **Implementation Scope:** FinOps Team | Procurement/Leadership
- **Prerequisites:** Long-term architectural stability

#### CE-11. Purchase Relational Database Service (RDS) RIs
- **What:** Leverage Cost Explorer's "Reservation Recommendations" for RDS to commit to 1- or 3-year usage of specific database instance types.
- **Why It Saves Money:** Reduces RDS compute costs by up to 69% compared to On-Demand rates.
- **Implementation Steps:**
  1. Open Cost Explorer -> Reservations -> Recommendations.
  2. Filter by Service: `RDS`.
  3. Purchase recommended Reserved Instances matching your deployed database engines and families.
- **Estimated Savings:** 30-69% on RDS compute
- **Risk Level:** Medium
- **Implementation Scope:** FinOps Team | Procurement/Leadership
- **Prerequisites:** Predictable RDS workloads

### 4. Architecture Changes

#### CE-12. Transition to S3 Intelligent-Tiering
- **What:** Analyze S3 usage in Cost Explorer (Service: `S3`, Usage Type: `StandardStorage`) to identify high storage costs, and move objects to S3 Intelligent-Tiering.
- **Why It Saves Money:** Standard storage is $0.023/GB. Intelligent-Tiering automatically moves rarely accessed data to Archive tiers ($0.004/GB or less), saving up to 80% on cold data.
- **Implementation Steps:**
  1. Identify high S3 Standard storage costs in Cost Explorer.
  2. Create an S3 Lifecycle Rule to transition objects to `INTELLIGENT_TIERING` after 0 days.
- **Estimated Savings:** 20-80% of S3 costs
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** S3 Lifecycle permissions

#### CE-13. Optimize Custom Billing View Queries
- **What:** Restructure internal querying logic to pull aggregated data instead of dynamically combining multiple Custom Billing Views on the fly.
- **Why It Saves Money:** CE API custom views cost $0.01 *per source included*. Aggregating logic locally prevents paying the multiplier on every single dashboard refresh.
- **Implementation Steps:**
  1. Audit Cost Explorer API usage using CloudTrail.
  2. Modify scripts to query the base data and perform joins/filtering in local memory or a database.
- **Estimated Savings:** 50-90% on Custom View CE costs
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps | FinOps Team
- **Prerequisites:** Scripting/Development capabilities

#### CE-14. Replace NAT Gateways with VPC Endpoints
- **What:** Use Cost Explorer to identify massive Data Processing fees on NAT Gateways caused by internal AWS API traffic (like S3 or DynamoDB), and replace them with VPC Endpoints.
- **Why It Saves Money:** NAT Gateways charge $0.045/GB for data processing. S3/DynamoDB Gateway Endpoints are 100% free, eliminating the data processing fee entirely for that traffic.
- **Implementation Steps:**
  1. Filter CE by Service: `VPC` and Usage Type: `NatGateway-Bytes`.
  2. If costs are high, create Gateway VPC Endpoints for S3 and DynamoDB in your VPC route tables.
- **Estimated Savings:** 50-90% of NAT Gateway transfer costs
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps (Networking)
- **Prerequisites:** VPC networking access

### 5. Scheduling & Auto-Scaling

#### CE-15. Detect Non-Prod Environments for Scheduling
- **What:** Group Cost Explorer costs by the `Environment` tag to spot Dev, QA, or Staging environments that generate costs 24/7.
- **Why It Saves Money:** Shutting down non-prod EC2/RDS instances for 12 hours a day and on weekends reduces their compute run-time from 168 hours/week to ~50 hours/week.
- **Implementation Steps:**
  1. Ensure Cost Allocation Tags (e.g., `env`) are activated in the Billing Console.
  2. Filter CE for `env: dev` and view hourly costs on weekends.
  3. Deploy AWS Instance Scheduler to automate start/stop schedules.
- **Estimated Savings:** 60-70% of non-prod compute costs
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** AWS Instance Scheduler deployed

#### CE-16. Identify Static Fleets for Auto-Scaling
- **What:** Use Cost Explorer to observe EC2 costs. If daily costs are a perfectly flat line regardless of traffic patterns, the environment is statically provisioned.
- **Why It Saves Money:** Flat-lined infrastructure must be provisioned for peak load. Auto-Scaling allows infrastructure to shrink during off-peak hours, saving money.
- **Implementation Steps:**
  1. Look at Daily EC2 costs in CE. A flat line indicates static provisioning.
  2. Implement EC2 Auto Scaling Groups (ASGs) tied to CPU or Request Count metrics.
- **Estimated Savings:** 20-40% of peak compute costs
- **Risk Level:** High (Requires infrastructure refactoring)
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Stateless application architecture

### 6. Pricing Model Optimization

#### CE-17. Shift Workloads to EC2 Spot Instances
- **What:** Identify heavy batch processing, CI/CD runners, or fault-tolerant workloads in Cost Explorer, and migrate them from On-Demand to EC2 Spot Instances.
- **Why It Saves Money:** Spot instances utilize spare AWS capacity and offer up to a 90% discount compared to On-Demand prices.
- **Implementation Steps:**
  1. Filter CE by Service: `EC2-Instances` and Usage Type: `BoxUsage` to find expensive worker fleets.
  2. Reconfigure ASGs or Kubernetes node groups to utilize Spot instances for interruptible tasks.
- **Estimated Savings:** 70-90% of targeted compute costs
- **Risk Level:** Medium (Instances can be interrupted)
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Workload must be fault-tolerant and interruptible

### 7. Network & Data Transfer Optimization

#### CE-18. Analyze and Reduce Cross-AZ Data Transfer
- **What:** Filter Cost Explorer for regional Data Transfer costs to identify architectures unnecessarily crossing Availability Zones.
- **Why It Saves Money:** AWS charges $0.01/GB for traffic crossing AZs in the same region. Keeping traffic within a single AZ (when high-availability isn't strictly needed for that microservice) eliminates this fee.
- **Implementation Steps:**
  1. Filter CE by Service: `EC2-Other` and Usage Type: `DataTransfer-Regional-Bytes`.
  2. Use VPC Flow Logs to trace the heavy traffic streams.
  3. Re-architect to colocate chatty microservices in the same AZ.
- **Estimated Savings:** 50-100% of inter-AZ transfer fees
- **Risk Level:** High (Can impact high-availability design)
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** VPC Flow Logs, architectural review

#### CE-19. Identify High Egress Traffic to the Internet
- **What:** Use Cost Explorer to monitor `DataTransfer-Out-Bytes` to spot spikes in internet egress costs.
- **Why It Saves Money:** AWS Internet Egress costs ~$0.09/GB. Offloading heavy static asset delivery to Amazon CloudFront lowers the fee to ~$0.085/GB (or much cheaper with custom pricing), and peering connections can reduce it further.
- **Implementation Steps:**
  1. Filter CE by Usage Type: `DataTransfer-Out-Bytes`.
  2. Identify the resources serving the traffic.
  3. Put a CDN (CloudFront) in front of the assets, or negotiate Private Pricing for heavy egress.
- **Estimated Savings:** 20-50% on Data Transfer Out
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps | Procurement
- **Prerequisites:** Application supports CDN caching

## Cross-Service Synergies
- **AWS Cost Explorer + AWS Organizations:** Cost Explorer provides centralized billing views across the entire organization when accessed from the management account.
- **AWS Cost Explorer + AWS Compute Optimizer:** Use Cost Explorer to identify *where* you are spending the most, and use Compute Optimizer to determine *how* to shrink those specific resources.
- **AWS Cost Explorer + Amazon Athena:** Migrating programmatic reporting to CUR + Athena eliminates the CE API bottleneck and API charges.

## Required Input Data for Real-World Analysis
### A. AWS CUR 2.0
The foundational source of truth for programmatic cost reporting. Replaces expensive API queries with raw S3 parquet files.
### B. CloudWatch Metrics
Required to augment Cost Explorer's billing data with actual utilization data (CPU, RAM) for Rightsizing.
### C. AWS Config / Trusted Advisor
Validates the current configuration of resources (like unattached EBS volumes) identified by Cost Explorer as wasteful.
### D. Company Policies
Determines risk appetite for utilizing Spot Instances, purchasing 3-year commitments, or enforcing off-hours scheduling.
### E. IaC (Optional)
Terraform or CloudFormation to implement the recommended architectural changes (e.g., deploying VPC Endpoints or Auto Scaling Groups).
