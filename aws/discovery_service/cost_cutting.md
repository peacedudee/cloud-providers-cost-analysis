## Executive Summary
AWS Application Discovery Service is a free tool designed to help enterprises plan their cloud migrations by automatically collecting data on their on-premises servers. While the service itself incurs $0.00 in direct charges, its real financial impact lies in how the collected data is used. Using Discovery Service effectively can slash migration costs, prevent post-migration architectural nightmares, and avoid cross-premises data transfer fees. Conversely, failing to clean up auxiliary resources (like S3 exports, Athena queries, and collector appliances) can create persistent, unnecessary cloud waste. This playbook provides strategies to leverage Discovery Service for massive overall cost avoidance while keeping its auxiliary costs at absolute zero.

## Cross-Service Synergies
- **AWS Migration Hub:** Discovery Service feeds natively into Migration Hub, creating a central repository for migration tracking at no extra cost.
- **Amazon S3 & Amazon Athena:** Used to store and query continuous data exports for custom reporting and right-sizing logic.
- **AWS Compute Optimizer:** Post-migration, right-sizing continues using EC2 metrics, picking up where Discovery Service left off.
- **Migration Evaluator:** Provides business case analysis using the data generated by Discovery Service.

#### 1. DISCOVERY-001: Implement S3 Lifecycle Rules on Discovery Export Buckets
- **What:** Automatically transition or delete exported CSV/JSON data from S3 after the migration analysis phase concludes.
- **Why It Saves Money:** S3 Standard storage costs $0.023/GB-month. While small, continuous data exports over months accumulate unnecessary storage costs post-migration.
- **Implementation Steps:**
  1. Identify the S3 bucket designated for Application Discovery Service data exports.
  2. Navigate to S3 bucket properties -> Lifecycle rules.
  3. Create a rule to expire/delete objects in the export prefix after 90 days.
- **Estimated Savings:** 1-5%
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** S3 Export feature enabled in Discovery Service.

#### 2. DISCOVERY-002: Decommission Agentless Collector OVA Post-Migration
- **What:** Shut down and delete the VMware vCenter Agentless Collector virtual appliances once data collection is complete.
- **Why It Saves Money:** Reclaims on-premises VMware compute, memory, and storage resources, delaying the need for on-premises hardware refresh or freeing capacity for other workloads.
- **Implementation Steps:**
  1. Verify in AWS Migration Hub that all necessary data collection has concluded.
  2. Log into the on-premises VMware vCenter.
  3. Power off the AWS Agentless Collector VMs.
  4. Delete the VMs from disk.
- **Estimated Savings:** 1-5%
- **Risk Level:** Low
- **Implementation Scope:** FinOps Team | Engineer/DevOps
- **Prerequisites:** Completion of the discovery phase.

#### 3. DISCOVERY-003: Uninstall Discovery Agents from Servers
- **What:** Remove the Application Discovery Agent from on-premises Windows and Linux servers post-migration or post-analysis.
- **Why It Saves Money:** Frees up minimal local CPU/RAM on source servers and stops continuous outbound HTTPS traffic, slightly reducing local network congestion and potential NAT processing fees.
- **Implementation Steps:**
  1. Identify servers where the agent was manually deployed.
  2. Use Configuration Management (Ansible, SCCM, Chef) to run uninstall scripts across the fleet.
  3. Validate removal in the AWS Migration Hub console (servers should show as inactive/stopped).
- **Estimated Savings:** <1%
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Agent-based discovery method used.

#### 4. DISCOVERY-004: Clean Up Athena Query Result Buckets
- **What:** Empty and delete the default S3 buckets used by Amazon Athena to store the results of custom queries run against discovery data.
- **Why It Saves Money:** Athena automatically saves query results to an S3 bucket (e.g., `aws-athena-query-results-*`). These files are often reviewed once and abandoned, incurring perpetual $0.023/GB-month fees.
- **Implementation Steps:**
  1. Identify the default Athena query results bucket.
  2. Implement an S3 Lifecycle rule to expire objects after 7-14 days.
  3. Delete historical result sets no longer needed.
- **Estimated Savings:** <1%
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Athena queries executed against discovery data.

#### 5. DISCOVERY-005: Right-Size Target EC2 Instances Using CPU/RAM Metrics
- **What:** Use Discovery Service performance data to size EC2 instances based on actual usage (e.g., 95th percentile CPU) rather than matching on-premises provisioned specs.
- **Why It Saves Money:** On-premises servers are typically heavily over-provisioned (e.g., 10-20% average utilization). Right-sizing prevents paying for unused EC2 compute capacity in the cloud.
- **Implementation Steps:**
  1. Export server metrics from Discovery Service to Amazon Athena.
  2. Query for average and peak (99th/95th percentile) CPU and RAM utilization over the collection period (ideally 14-30 days).
  3. Map these metrics to the smallest capable modern EC2 instance family (e.g., AWS Graviton, T-series, or M-series).
- **Estimated Savings:** 30-50%
- **Risk Level:** Medium
- **Implementation Scope:** FinOps Team | Engineer/DevOps
- **Prerequisites:** Data collection ran long enough to capture peak business cycles.

#### 6. DISCOVERY-006: Identify and Exclude Zombie Servers from Migration
- **What:** Use Discovery data to identify servers with near-zero CPU usage, memory churn, and network connections, and decommission them entirely instead of migrating them.
- **Why It Saves Money:** Avoids 100% of the migration effort, target compute, target storage (EBS), and OS licensing costs for applications that are no longer used.
- **Implementation Steps:**
  1. Analyze Discovery Service performance metrics for servers with < 2% CPU utilization and 0 inbound network connections over 30 days.
  2. Interview app owners to verify the server is a "zombie."
  3. Schedule the server for retirement on-premises.
- **Estimated Savings:** 5-15%
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps | FinOps Team
- **Prerequisites:** Minimum 30 days of data collection.

#### 7. DISCOVERY-007: Right-Size Local Resources for Agentless Collector
- **What:** Allocate only the necessary vCPU and RAM to the Agentless Collector OVA on-premises.
- **Why It Saves Money:** Prevents the VMware cluster from unnecessarily allocating premium compute resources to the data collection appliance.
- **Implementation Steps:**
  1. Review the AWS sizing guidelines for the Agentless Collector.
  2. Adjust the VM allocation in vCenter (typically 4 vCPUs and 16 GB RAM is sufficient for up to 10,000 VMs).
  3. Avoid over-provisioning beyond these specs.
- **Estimated Savings:** <1%
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** VMware vCenter access.

#### 8. DISCOVERY-008: Leverage Data for Compute Savings Plan Forecasting
- **What:** Use the right-sized target EC2 baseline generated from Discovery Service to confidently purchase Compute Savings Plans immediately upon migration.
- **Why It Saves Money:** Reduces target EC2 compute rates by up to 66% compared to On-Demand by establishing an accurate baseline for a 1- or 3-year commitment.
- **Implementation Steps:**
  1. Finalize the right-sized target EC2 inventory using Discovery data.
  2. Calculate the steady-state hourly spend requirement.
  3. Purchase a Compute Savings Plan covering 70-80% of that steady-state baseline just before or during the migration cutover.
- **Estimated Savings:** 20-30%
- **Risk Level:** Low
- **Implementation Scope:** FinOps Team | Procurement/Leadership
- **Prerequisites:** Right-sizing analysis (DISCOVERY-005) completed.

#### 9. DISCOVERY-009: Maximize AWS MAP Credits with Accurate Baselines
- **What:** Use comprehensive Discovery data to ensure every eligible server is accounted for when applying for AWS Migration Acceleration Program (MAP) funding.
- **Why It Saves Money:** AWS MAP provides significant cash and credit incentives (up to 25% of annual recurring revenue) for migrating workloads, but the incentive is based on accurate pre-migration baselines.
- **Implementation Steps:**
  1. Ensure Discovery Service agents/collectors cover 100% of the on-premises footprint.
  2. Generate a comprehensive inventory report from Migration Hub.
  3. Provide this exhaustive list to your AWS Account Team or Partner to maximize the MAP funding calculation.
- **Estimated Savings:** 10-25%
- **Risk Level:** Low
- **Implementation Scope:** FinOps Team | Procurement/Leadership
- **Prerequisites:** Enterprise Agreement and MAP eligibility.

#### 10. DISCOVERY-010: Map Network Dependencies to Prevent Egress Costs
- **What:** Use the network connection data captured by Discovery Service to group tightly coupled applications into the same migration "wave."
- **Why It Saves Money:** If an app server is migrated to AWS but its dependent database remains on-premises for months, every database query incurs cross-environment AWS Data Egress fees ($0.09/GB). Grouping prevents this hybrid latency and cost.
- **Implementation Steps:**
  1. Query the network dependency data collected by Discovery Agents or Agentless Collector.
  2. Identify high-bandwidth, high-frequency TCP/UDP connections between servers.
  3. Bundle tightly coupled servers into a single migration wave in Migration Hub.
- **Estimated Savings:** 10-40%
- **Risk Level:** High
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Agent-based discovery or VMware integration for network flow data.

#### 11. DISCOVERY-011: Consolidate Licenses by Analyzing Running Processes
- **What:** Analyze the running processes collected by Discovery Agents to identify instances of expensive commercial software (e.g., MS SQL, Oracle) and consolidate them into fewer, denser EC2 instances or RDS.
- **Why It Saves Money:** Cloud BYOL (Bring Your Own License) for commercial databases is heavily tied to vCPU counts. Identifying and consolidating sparsely used database instances saves massive licensing fees.
- **Implementation Steps:**
  1. Export the "System Performance" and "Processes" data from Migration Hub.
  2. Filter for known expensive process executables (`sqlservr.exe`, `oracle`).
  3. Plan migrations to consolidate these databases into higher-density RDS instances.
- **Estimated Savings:** 10-20%
- **Risk Level:** Medium
- **Implementation Scope:** FinOps Team | Engineer/DevOps
- **Prerequisites:** Discovery Agents installed (Agentless collector does not capture OS-level processes).

#### 12. DISCOVERY-012: Prefer Agentless Collector Over Agent-Based
- **What:** Where possible (i.e., in VMware environments), use the Agentless Collector instead of installing agents on every single OS.
- **Why It Saves Money:** Deploying, troubleshooting, and maintaining agents across thousands of servers costs significant engineering hours. Agentless collection requires managing a single appliance, massively reducing operational labor costs.
- **Implementation Steps:**
  1. Assess the on-premises environment. If >80% virtualized on VMware, default to Agentless Collector.
  2. Only deploy the OS-based Discovery Agent on physical bare-metal servers or non-VMware hypervisors.
- **Estimated Savings:** 1-5%
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** VMware vCenter environment.

#### 13. DISCOVERY-013: Modernize Workloads Identified as Container Candidates
- **What:** Use OS and Process data from Discovery Service to identify legacy workloads that can be easily refactored into Docker containers.
- **Why It Saves Money:** Moving from full EC2 instances to Amazon ECS on Fargate, or eliminating Windows OS licenses by modernizing to Linux containers, dramatically reduces compute and licensing overhead.
- **Implementation Steps:**
  1. Filter Discovery data for standard, stateless web server processes.
  2. Tag these servers in Migration Hub for "Refactor/Modernize" rather than "Rehost".
  3. Migrate directly to AWS AppRunner or ECS.
- **Estimated Savings:** 20-40%
- **Risk Level:** High
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Discovery Agents deployed to capture process details.

#### 14. DISCOVERY-014: Limit Duration of Discovery Data Collection
- **What:** Stop data collection once a statistically significant baseline (usually 14-30 days) is established, rather than running it indefinitely.
- **Why It Saves Money:** Indefinite data collection generates massive CSV/JSON exports into S3, inflating storage costs and complicating Athena queries with overly massive datasets without adding new insights.
- **Implementation Steps:**
  1. Monitor data collection in AWS Migration Hub.
  2. Once 30 days of continuous business-cycle data is captured, stop the Agent/Collector.
  3. Proceed with the right-sizing and dependency analysis.
- **Estimated Savings:** <1%
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** None.

#### 15. DISCOVERY-015: Optimize Athena Query Scanning Costs
- **What:** If performing heavy custom analysis on exported Discovery data, write efficient SQL queries that limit data scanning, as Athena charges $5.00 per TB scanned.
- **Why It Saves Money:** Running unstructured `SELECT *` queries across months of performance metrics can unnecessarily scan terabytes of data.
- **Implementation Steps:**
  1. Only query the necessary columns (Athena is columnar).
  2. Include date filters in the `WHERE` clause to restrict the query to specific timeframes.
  3. Create materialized views for frequently accessed aggregation data.
- **Estimated Savings:** 1-5%
- **Risk Level:** Low
- **Implementation Scope:** FinOps Team | Engineer/DevOps
- **Prerequisites:** Continuous export to S3 enabled.

#### 16. DISCOVERY-016: Route Agent Traffic via Direct Connect
- **What:** Ensure that outbound traffic from on-premises Discovery Agents to the AWS endpoint traverses an existing AWS Direct Connect or VPN, rather than egressing out to the public internet.
- **Why It Saves Money:** If agents go out to the public internet, they may pass through expensive on-premises enterprise firewalls/proxies. Or, if they pass through a cloud-hosted NAT Gateway for inspection, it incurs $0.045/GB processing fees.
- **Implementation Steps:**
  1. Configure network routes to ensure traffic destined for AWS Application Discovery Service endpoints uses the Direct Connect private VIF.
  2. Set up AWS PrivateLink (VPC Endpoints) for Discovery Service if routing through a transit VPC.
- **Estimated Savings:** 1-5%
- **Risk Level:** Medium
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** AWS Direct Connect or Site-to-Site VPN in place.

#### 17. DISCOVERY-017: Filter Irrelevant Subnets from Agentless Collector
- **What:** Configure the Agentless Collector to only discover and monitor specific VMware clusters or resource pools containing in-scope workloads.
- **Why It Saves Money:** Prevents collecting metrics on VDI environments, development sandboxes, or infrastructure management VMs that have no intention of migrating to AWS, keeping the dataset clean and reducing S3 export sizes.
- **Implementation Steps:**
  1. Access the Agentless Collector local UI.
  2. In the vCenter configuration, explicitly select the target clusters/folders to monitor.
  3. Exclude non-migrating resource pools.
- **Estimated Savings:** <1%
- **Risk Level:** Low
- **Implementation Scope:** Engineer/DevOps
- **Prerequisites:** Agentless Collector deployed.
