# Dataplex Cost Optimization & Research

Google Cloud Dataplex is an intelligent data fabric that helps organizations unify, discover, manage, and govern data across Google Cloud Storage, BigQuery, and other data sources. While the logical organization of data into Lakes, Zones, and Assets is free, Dataplex charges for metadata harvesting, data profiling, data quality scans, and serverless Spark execution.

---

## 1. Dataplex Billing Components

Dataplex charges are based on two primary dimensions:
1. **Data Compute Units (DCUs):** Billed per DCU-hour (approx. $0.089 per DCU-hour). DCUs represent the CPU, RAM, and network capacity used to run:
   * **Metadata Ingestion/Discovery:** Automated crawler jobs that discover schema changes in GCS and BigQuery.
   * **Data Quality (DQ) Scans:** Running declarative rules to check data consistency.
   * **Data Profiling:** Visual statistics calculations on data distributions.
2. **Serverless Spark Tasks:** Billed per DCU-hour when running custom Spark jobs through Dataplex.
3. **Underlying Data Scans:** Running DQ or profiling scans on BigQuery tables triggers standard **BigQuery query scan charges** ($5.00/TB scanned) or Spanner/GCS read operations.

---

## 2. Core Cost-Optimization Levers

### A. Restrict Data Quality Scans to Incremental Data
* **The Waste:** Running a weekly Data Quality scan that processes the entire historical contents of a 50 TB sales table. You pay high DCU fees for Spark and high BigQuery query scan fees ($250 per run).
* **The Solution:** Use **Incremental Scans**.
* **Action:** In the Dataplex Data Quality job settings, configure the scan to run incrementally. Specify a partition column (e.g. `sale_date`) and filter the scan to process only the latest partition (e.g. `ds = CURRENT_DATE()`).

### B. Adjust Metadata Discovery Schedule
Dataplex crawlers scan GCS buckets to register metadata tables.
* **The Cost Trap:** Leaving crawlers configured to run "Continuously" or "Hourly" on highly volatile buckets.
* **Action:** Change the discovery schedule setting to **Daily** or **Weekly**, or configure it to run on-demand via CI/CD triggers when new batch datasets are published.

### C. Optimize Data Profiling Intervals
* Data profiling provides valuable stats but does not need to run on every micro-batch update.
* **Action:** Limit data profiling tasks to run weekly on core production datasets. Disable profiling on staging and development lakes entirely.

---

## 3. Dataplex Audit Checklist

1. [ ] **Incremental Scan Verification:** Confirm all active Dataplex Data Quality jobs are configured with incremental scan filters.
2. [ ] **Crawler Schedule Review:** Audit metadata crawlers. Adjust schedules from hourly/continuous to daily/weekly.
3. [ ] **Development Lake Pruning:** Ensure non-prod lakes do not have scheduled DQ or profiling tasks running automatically.
4. [ ] **BigQuery Query Auditing:** Check BigQuery logs to identify query charges generated by the `dataplex-service-account` and optimize associated scan ranges.
